# Contributing to QML Agent Lab

Thank you for helping improve QML Agent Lab.

This project is a lightweight research and development toolkit for agent-assisted quantum machine learning experiments. Contributions should make experiments more reproducible, more honest, more useful, or easier for humans and AI agents to extend.

## Core Contribution Principles

1. Every QML experiment must include a classical baseline.
2. Claims must be proportional to evidence.
3. Educational examples must be labeled as educational.
4. Results must document limitations.
5. Scripts should be small, readable, and reproducible.
6. New dependencies should be justified.
7. Do not claim quantum advantage without rigorous, controlled evidence.

## Preferred Contribution Types

Good first contributions include:

- Improving documentation clarity
- Adding small classical baselines
- Adding small QML comparison examples
- Improving result templates
- Adding benchmark reports under `results/`
- Adding reproducibility metadata
- Updating deprecated API usage
- Improving error handling in examples

## Repository Structure

Use the existing structure:

```text
agents/      Agent role documents
templates/   Reusable experiment and report templates
docs/        Tooling maps and project documentation
examples/    Runnable educational scripts
results/     Measured benchmark reports and machine-readable records
```

## Experiment Contribution Checklist

Before adding a new experiment, confirm:

- [ ] A classical baseline is included.
- [ ] The dataset is documented.
- [ ] The train/test split or validation method is documented.
- [ ] Random seeds are documented.
- [ ] QML framework, backend, qubits, shots, and circuit depth are documented where applicable.
- [ ] Metrics are appropriate for the task.
- [ ] Limitations are stated clearly.
- [ ] No unsupported quantum advantage claim is made.

## Code Style

Python examples should:

- Use `main()`.
- Use `if __name__ == "__main__":`.
- Handle missing imports gracefully.
- Use local toy datasets unless otherwise justified.
- Avoid cloud services by default.
- Avoid long training loops in starter examples.
- Prefer clarity over abstraction.

## Results Style

Measured outputs should go in `results/`.

For each benchmark, prefer both:

- a Markdown report for human review, and
- a JSON file for machine-readable agent workflows.

Results should state whether they are:

- generated by script,
- manually verified,
- CI-verified,
- single-seed,
- multi-seed,
- simulated, or
- hardware-backed.

## Documentation Style

Documentation should be concise, specific, and honest.

Use language such as:

- "educational benchmark"
- "toy dataset"
- "single-seed result"
- "requires further validation"
- "not evidence of quantum advantage"

Avoid language such as:

- "breakthrough"
- "proves"
- "revolutionary"
- "beats classical ML" without context
- "quantum advantage" without rigorous support

## Pull Request Checklist

Before opening a pull request:

- [ ] The change is scoped and understandable.
- [ ] README or docs are updated if behavior changed.
- [ ] Changelog is updated if user-facing content changed.
- [ ] Examples still fail gracefully when dependencies are missing.
- [ ] New results include limitations.
- [ ] Claims are modest and evidence-based.

## Agent-Aware Contributions

This repo is designed to be used by AI coding agents.

When adding docs or examples, make them easy for agents to parse:

- Use clear headings.
- Use checklists.
- Use explicit metadata.
- Keep file names descriptive.
- Prefer structured Markdown and JSON where appropriate.

## Final Note

The goal is not to make QML look impressive by default.

The goal is to make QML experiments easier to design, compare, reproduce, and evaluate honestly.
